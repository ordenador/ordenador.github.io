# 003 — i18n (ES/EN)

## Problem

Content is English-only while the page declares `lang="es"`; Chilean recruiters get English, and there is no way to offer both. UI labels ("Experience", "Download PDF") are hardcoded in components.

## Scope

- Two locales: `en` (current content) and `es` (translation of `resume.js` + UI labels).
- Data layer: split `src/data/resume.js` into per-locale data (`resume.en.js`, `resume.es.js`) sharing one shape; UI strings into a small `labels` object per locale. No i18n library — a `useLocale()` context + object lookup covers 2 locales (YAGNI on react-i18next).
- Language switcher in the header (EN/ES toggle); choice persisted in `localStorage`; default from `navigator.language`; `<html lang>` updated on switch.
- PDF per locale: `mario_faundez_cv.en.pdf` and `mario_faundez_cv.es.pdf`, both generated in CI (spec 002); download button links to the active locale's PDF. Keep `mario_faundez_cv.pdf` as alias of EN for existing links.
- SEO: `hreflang` alternates via query param or hash-free approach (`?lang=es`), meta description per locale.

## Non-goals

- No third language, no locale-specific routing (`/es/` paths), no translation tooling.

## Acceptance

- Toggle switches every visible string and the PDF link; reload keeps the choice.
- Both PDFs generated by `make generate-pdf` and by CI; content matches the active data files.
- Old URL `…/mario_faundez_cv.pdf` still resolves.
